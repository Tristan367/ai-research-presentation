The Landscape

Why now is the right time to invest in local AI infrastructure.

🧠

Openweight Models Are Production-Ready

Models like Llama 3.1, Qwen 2.5, and Mistral deliver performance that rivals proprietary APIs. They run entirely offline, handle complex reasoning, and can be finetuned for domain-specific tasks. The gap between open and closed models has narrowed dramatically.

⚡

Hardware Is Accessible

Consumer GPUs like the RTX 3090 and 4090 provide massive compute at a fraction of enterprise pricing. A single RTX 3090 can run 27B models like Qwen3.6-27B and Gemma 26B interactively — and they are fast. The break-even point between cloud and local ownership is now measured in months, not years.

🛠

Tooling Is Mature

Ollama, vLLM, Unsloth, and a rich ecosystem make it possible for a single developer to deploy production-grade AI systems. Finetuning, SQL tooling and agentic workflows are all accessible without a team of ML engineers.

A cloud coding agent (Cursor Composer) was then brought in to validate pricing against vendor sources, apply large multi-section edits, and add the independent model comparison below.

Cloud vs. Local

An honest assessment of both approaches.

Cloud GPUs

Pros

  • Zero upfront capital expenditure
  • Instant access to H100 / A100 hardware
  • Elastic scalability — spin up/down as needed
  • No hardware maintenance or cooling concerns
  • Ideal for R&D, experimentation, and prototyping

Cons

  • Ongoing costs accumulate fast (~$2,100–2,400/mo for single H100 24/7 at $2.89–3.29/hr)
  • Data leaves your premises — privacy concerns
  • Network latency for inference
  • GPU availability can be unreliable at peak demand
  • No tangible asset at the end — pure expense

Local Infrastructure

Pros

  • One-time investment — own the hardware
  • Complete data privacy — everything stays on-premises
  • Zero network latency for inference
  • Always available — no queueing or outages
  • Unlimited experimentation without hourly anxiety
  • Resale value at end of life

Cons

  • Upfront capital required ($3,000–5,000 for a 4090 rig)
  • Physical space, power, and cooling requirements
  • Hardware maintenance and driver updates
  • Fixed capacity — cannot elastically scale up
  • Hardware depreciates over time
Recommendation: Start with cloud for the POC phase to validate the approach quickly and with minimal risk. Transition to local infrastructure for production deployment to reduce long-term costs and ensure data sovereignty.

Why SQL Tooling

For structured sales data, the model generates SQL queries against the live database — no chunking, no embeddings, no pipeline.

How It Works

  • User asks a question in plain English
  • Model receives the database schema in its system prompt
  • Model generates a SQL query against the live data
  • Query is validated and executed (read-only, SELECT only)
  • Results are formatted into a natural language answer
One Model, One Query, One Answer: Sales data is tabular — transaction IDs, dates, amounts, rep IDs. There is no reason to chunk it, embed it, or build a pipeline around it. A single 7B model with schema awareness answers any aggregate question with perfect accuracy. No finetuning required.

POC Proposal

Goal: An LLM that answers questions about sales statistics using direct SQL queries — no vector DB, no embeddings.

Approach: Single Model + SQL Tooling

A single 7B model with schema awareness generates SQL queries against the live sales database — translating natural language to SQL, executing it, and formatting the result in plain English. The model maps natural language to SQL, executes it, and formats the results in plain English.

Why SQL Tooling?

  • Sales data is structured: Tables, not paragraphs
  • Aggregations matter: SUM, AVG, COUNT, GROUP BY — not approximate retrieval
  • Instant updates: No re-indexing needed when data changes
  • Exactly correct: SQL returns precise numbers, not "relevant chunks"
  • Minimal stack: No embedding model, no vector DB to manage

Model Choice

  • Llama 3.1 8B: Strong SQL generation, 4.9GB at Q4, fits on RTX 3090
  • Qwen2.5 7B: Excellent reasoning, competitive SQL performance
  • Both fit on RTX 3090: 24GB VRAM sufficient for inference
  • Schema in system prompt: Table definitions provided at inference time
Recommended Stack: Ollama for inference · SQLite/Postgres for the database · Schema-aware system prompts · Simple Python wrapper to execute generated SQL safely

Deployment Strategy

Two environments: one for building, one for the team.

Dev Pod

For building, finetuning, and the POC

  • Runpod GPU Pod (RTX 3090 ~$0.34/hr)
  • Always warm — model loaded, zero latency
  • Unsloth Studio for interactive finetuning and chat
  • SSH and Jupyter access for debugging
  • Persistent storage for models and datasets

Serverless

For team access in production

  • Runpod Serverless — auto-scaling API endpoint
  • Min workers: 1 (always warm, no cold start for first user)
  • Max workers: 5 (scales up during events)
  • Scales to minimum when traffic drops
  • API endpoint — connect any frontend (web app, Slack bot, etc.)
Min workers = 1 means one GPU is always running. The first person to ask a question gets an instant response — no cold start. When concurrent traffic exceeds what one GPU can handle, a second worker spins up (~20-60s cold start for additional workers). Under event load it scales to the max. When traffic dies down, it scales back to the minimum. The idle worker costs ~$250/month — worth it to eliminate cold starts for a team tool. With min workers = 0, you pay nothing when idle but the first request each time takes 20-60s to load the model.

How It Works: The system prompt includes the full database schema (table names, column names, types, relationships). The model generates a SQL query, which is validated and executed against the live database. Results are fed back to the model for natural language formatting. No embeddings. No chunking. No vector DB.

Schema Awareness

The model receives table definitions in its system prompt: CREATE TABLE statements, column types, foreign keys, and a few example rows. This gives it full context to generate correct SQL.

Safe Execution

Generated SQL runs through a read-only validation layer. Only SELECT statements are permitted. A simple Python wrapper catches syntax errors and retries with error feedback.

Natural Language Output

Raw SQL results (rows of numbers/text) are passed back to the LLM with a formatting prompt. The model turns "[(42, 15000.00)]" into "Rep X closed 42 deals for $15,000 total."

Anticipated Questions

01

"How does the SQL generation handle edge cases or weird questions?"

The model is given schema + example rows in the system prompt. Invalid SQL is caught by a validation layer and the model retries with the error message. Hallucinated column names get corrected on retry. For truly unanswerable questions, it says "I don't have that data."

02

"What if someone asks a question that needs data we don't have?"

The model generates SQL against the actual schema it was given. If a requested column or table doesn't exist, the SQL fails validation, and the model informs the user. No hallucinated data — it works with what's actually there.

03

"Is this secure? Could someone inject SQL or leak data?"

Only SELECT statements are permitted — no INSERT, UPDATE, DELETE, or DROP. The Python wrapper enforces read-only mode. The model runs locally, so data never leaves the building. This is actually more secure than cloud API solutions where prompts and data are logged by the provider.

04

"What happens when the database schema changes?"

Update the system prompt. That's it. No retraining. No re-indexing. No downtime. The model adapts to schema changes instantly because it reads the schema at inference time.

05

"How long until this is actually useful for the team?"

Week 1: working prototype on cloud. Week 2-3: iterating with real data. Week 4: demo. Month 2: running locally on dedicated hardware. The POC is measured in days, not months.

06

"What's the fallback if the local GPU dies?"

Two options: (1) Keep a cloud GPU on standby — the pipeline is identical, just swap the endpoint. (2) Buy two 3090s for redundancy — at $1,180 each, that's still cheaper than 2 months of cloud H100. The architecture is portable between cloud and local.

07

"Does the model need finetuning to be good at SQL?"

Not for the POC. Off-the-shelf Llama 3.1 8B and Qwen2.5 7B are already very capable at generating correct SQL from schema context. Finetuning would only matter if we hit specific edge cases with our particular schema or query patterns. Even then, a LoRA adapter on a 7B model takes hours, not days — it's an optimization, not a prerequisite.

08

"What if the model generates a slow query?"

We set a query timeout in the wrapper and can add EXPLAIN ANALYZE feedback to help the model self-correct. Database-side indexing handles the rest. The model generally produces reasonable queries because it sees the schema and understands the data shape.

09

"Is auto-scaling a small model on cloud GPUs better long-term? Especially with variable traffic — sometimes nobody's on, sometimes thousands during events?"

This changes the math completely. If traffic is spiky — long periods of zero usage punctuated by bursts — buying on-prem hardware is wasteful. Your GPU sits idle most of the time. With Runpod Serverless or Hyperbolic, the model scales to zero when nobody's using it (you pay nothing) and spins up instantly during events (you pay for GPU-seconds only). For a small 8-9B model that runs comfortably on a single 16-24GB GPU, the monthly cloud cost at moderate usage is $50-200/month. Buying a $1,180 3090 would take 6-12 months to break even, and that's only if it runs 24/7. For variable traffic, cloud auto-scaling wins on every axis: zero waste, zero maintenance, zero upfront. The strategy should be: start in the cloud, buy dedicated hardware only if and when utilization justifies it.

10

"Why not just use AWS? Isn't that the standard cloud for AI?"

AWS is the standard for enterprise infrastructure, but for GPU compute it charges a massive premium. An AWS p4d instance (8x A100) runs ~$33/hr — and you're forced to pay for the entire EC2 stack (VPC, EBS, networking, support) whether you use it or not. A single A100 on Runpod or Hyperbolic is ~$1.50-2.00/hr. Same GPU, same CUDA, 5-10x cheaper. AWS makes sense when you need thousands of GPUs, enterprise compliance, or tight S3/SageMaker integration. For a single-GPU POC, it's the wrong tool — you're paying enterprise rates for a hobby project. Specialized GPU providers (Runpod, CoreWeave, Hyperbolic) strip out the overhead and charge for the GPU alone. Same result, fraction of the cost.

11

"Can non-technical team members use this?"

That's the whole point. They type a question in English, get an answer in English. No SQL knowledge required. The model handles translation, the database handles computation, the user just asks.

12

"What does this actually do for our organization? How does it help retain members or grow revenue?"

The direct ROI is staff efficiency — instead of someone manually digging through spreadsheets or databases to answer "how many signups did we get this month?" or "what's the average donation per member?", anyone on the team asks in English and gets an answer in seconds. The indirect value is bigger: faster answers mean better member support, better support means happier members, happier members means better retention. And for a small organization, the analytics insights that used to require "ask the person who knows SQL" become accessible to everyone — event coordinators, membership coordinators, leadership. That alone can surface trends and opportunities that would otherwise stay buried.
