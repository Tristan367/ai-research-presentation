The Landscape

Why now is the right time for WWG to invest in local AI infrastructure.

🧠

Openweight Models Are Production-Ready

Models like Llama 3.1, Qwen 2.5, and Mistral deliver performance that rivals proprietary APIs. They run entirely offline, handle complex reasoning, and can be finetuned for domain-specific tasks. The gap between open and closed models has narrowed dramatically.

⚡

Hardware Is Accessible

Consumer GPUs like the RTX 3090 and 4090 provide massive compute at a fraction of enterprise pricing. A single RTX 3090 can run 7B–13B parameter models interactively. The break-even point between cloud and local ownership is now measured in months, not years.

🛠

Tooling Is Mature

Ollama, vLLM, LangChain, Unsloth, and a rich ecosystem make it possible for a single developer to deploy production-grade AI systems. Finetuning, RAG pipelines, and vector databases are all accessible without a team of ML engineers.

Proof of Concept: This presentation itself was researched, written, and assembled by Qwen3.6-27B running locally on a single RTX 3090 — demonstrating that meaningful AI work is possible on accessible hardware.

Lessons from AI Builders

What recent AI startup founders and practitioners are saying about building in 2025–2026.

💡

Taste & Judgment Trump Coding

As WIRED reported (Oct 2025), building software with AI is now so fast that the critical skill is no longer coding—it’s taste: knowing what to build. A customer bug or feature request can go from email to deployed feature in 15 minutes. The hard part is deciding whether you should build it at all. This means our POC should focus on solving a real WWG problem, not demonstrating technical prowess.

📊

Data Is the Bottleneck, Not Compute

Every source agrees: the hardest part of an AI project is gathering, curating, and managing data—not training the model. For WWG, this means getting quality sales data from Bret is the single most important next step. Without good data, no amount of GPU power will produce useful results.

🎯

Ship Fast, Iterate Faster

AI founders report being able to run countless experiments because the cost of trying a new approach has dropped to near zero. Code is disposable. The strategy is to parallel-universe yourself into multiple approaches, then collapse into the one that works. Our phased roadmap embodies this: start with a cloud POC in 2 weeks, pivot based on real feedback, then invest in local hardware only after proving value.

🤖

Validate Before You Build

thoughtbot’s framework (updated Dec 2023) emphasizes testing problem-solution fit before investing in AI development. Start with a simple prototype using existing models and human-in-the-loop validation. Only build custom AI when no off-the-shelf solution works. For WWG, this means: prototype the RAG pipeline with an existing model first, then decide if finetuning is worth it.

🔧

Find Your First Customer Fast

Multiple AI startup founders said the most time-consuming part of their work was finding their first human customer—not building the product. For our internal POC, Bret and the sales team are our “first customers.” Getting them to use the tool early and give feedback is the priority over perfecting the model.

🚀

Go After Hard Problems

WIRED (Oct 2025) found that the easiest AI ideas are now saturated. Founders who succeed push into harder, domain-specific problems. WWG’s complex sales data—with its proprietary knowledge, seasonal patterns, and nuanced relationships—is exactly the kind of hard, defensible problem that AI can solve but that generic models cannot.

Cloud vs. Local

An honest assessment of both approaches.

Cloud GPUs

Pros

  • Zero upfront capital expenditure
  • Instant access to H100 / A100 hardware
  • Elastic scalability — spin up/down as needed
  • No hardware maintenance or cooling concerns
  • Ideal for R&D, experimentation, and prototyping

Cons

  • Ongoing costs accumulate fast ($2,080/mo for H100 24/7)
  • Data leaves your premises — privacy concerns
  • Network latency for inference
  • GPU availability can be unreliable at peak demand
  • No tangible asset at the end — pure expense

Local Infrastructure

Pros

  • One-time investment — own the hardware
  • Complete data privacy — everything stays on-premises
  • Zero network latency for inference
  • Always available — no queueing or outages
  • Unlimited experimentation without hourly anxiety
  • Resale value at end of life

Cons

  • Upfront capital required ($3,000–5,000 for a 4090 rig)
  • Physical space, power, and cooling requirements
  • Hardware maintenance and driver updates
  • Fixed capacity — cannot elastically scale up
  • Hardware depreciates over time
Recommendation: Start with cloud for the POC phase to validate the approach quickly and with minimal risk. Transition to local infrastructure for production deployment to reduce long-term costs and ensure data sovereignty.

Training from Scratch vs. Finetuning

WWG should not train a model from scratch. Here is why finetuning is the right approach.

Training from Scratch

  • Hardware: Thousands of H100 GPU-hours
  • Cost: $100,000 – $10,000,000+
  • Time: Weeks to months
  • Data: Trillions of tokens
  • Team: Team of ML engineers and researchers
  • Output: General-purpose model
Verdict: NOT VIABLE

Finetuning

  • Hardware: Single RTX 3090 (24GB VRAM)
  • Cost: $0 – $50 (cloud GPU hours)
  • Time: Hours to days
  • Data: 100 – 5,000 labeled examples
  • Team: One person
  • Output: Domain-specific expert model
Verdict: RECOMMENDED
Why Finetuning Wins: Techniques like LoRA (Low-Rank Adaptation) freeze the base model and train only 0.1–1% of its parameters. This means you can adapt a 7B–70B parameter model to a specific domain using a single consumer GPU, in hours, for virtually no cost. The base model general knowledge is preserved while new domain expertise is layered on top.

LoRA

Low-Rank Adaptation — injects small trainable matrices into the model. Trains only 0.1–1% of parameters. Outputs are small adapter files (100MB–2GB) that can be swapped between tasks.

QLoRA

Quantized LoRA — combines 4-bit quantization with LoRA. Reduces VRAM requirements by 3x, enabling finetuning of 70B models on a single 24GB GPU with minimal quality loss.

Tools

Unsloth: 2x faster finetuning, 60% less VRAM. axolotl: YAML-configured finetuning. TRL: Hugging Face Transformers Reinforcement Learning library for DPO and RLHF.

WWG POC Proposal

Goal: An LLM that answers questions about WWG sales statistics accurately and quickly.

Approach: RAG + Finetuned Small Model

Retrieval-Augmented Generation combines a finetuned base model with a vector database of WWG data. The model gets domain knowledge from finetuning and factual accuracy from real-time data retrieval.

Why RAG Instead of Full Finetuning?

  • Large/complex dataset: Sales data changes constantly
  • Don stuff everything into weights: Query data at inference time instead
  • Accuracy: Grounded answers from actual data, not memorization
  • Updatability: New data just needs re-indexing, not retraining
  • Smaller model needed: 7B-8B model is sufficient when augmented with data

Model Choice

  • Llama 3.1 8B: 4.9GB at Q4, excellent reasoning, fits on RTX 3090
  • Qwen2.5 7B: Compact, strong performance, Apache 2.0 license
  • Both fit on RTX 3090: 24GB VRAM is sufficient for inference
  • vLLM serving: PagedAttention, continuous batching, 200+ model architectures
Recommended Stack: Ollama or vLLM for inference · LangChain or LlamaIndex for RAG orchestration · ChromaDB for vector storage · Sentence Transformers for embeddings

Orchestration: LangChain handles the entire pipeline — query processing, embedding, retrieval, prompt assembly, and response formatting. The LLM never sees raw data; it receives a prompt containing the original question plus the most relevant retrieved context chunks.

Text Preprocessing

WWG sales data is chunked into overlapping segments (512 tokens, 64-token overlap). Each chunk is embedded using a pre-trained sentence transformer model.

Vector Storage

Embeddings are stored in ChromaDB with metadata (date, region, rep name). At query time, the top-k most similar chunks are retrieved by cosine similarity.

Prompt Assembly

The retrieved context is injected into a system prompt: "You are a WWG sales analytics assistant. Answer using only the provided context data." The LLM generates a grounded response.
